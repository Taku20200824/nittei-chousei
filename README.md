## 目的
イベントの候補日時に対し、参加者が各候補へ **○（参加できる）/ ✕（参加できない）** だけを回答し、全員の回答を1つの表で共有・集計するためのツール。URL を送るだけで、相手も同じ表を見てリアルタイムに回答できます。

## 主な機能
- タイトル・メモ・候補日時（日付＋任意の時間）でイベントを作成
- 参加者は名前＋各候補に **○ / ✕** で回答（未回答は「－」）
- 全回答を表で集計し、**○の合計** と **最有力候補** を自動ハイライト
- 作成者（管理者）は候補の追加・削除・編集、全回答の編集・削除が可能
- 各参加者は **自分の回答だけ** を編集・削除可能
- **リアルタイム同期**（Firebase の `onValue` リスナー。ポーリングなし）
- ダークモード対応・モバイル対応（レスポンシブ）
- この端末で作成したイベントの履歴（`localStorage`）
- オフライン／同期済み／権限なし／通信エラー等の状態表示

## 使い方
1. トップページで **イベントを作成**（タイトルと候補日時を入力）。作成した端末が自動的にそのイベントの「管理者」になります。
2. **「🔗 共有リンクをコピー」** で URL を相手に送ります。
3. 相手はリンクを開き、名前と各候補の ○ / ✕ を入力して **登録**。回答は全員の画面に数秒以内で反映されます。
4. 管理者は候補の編集や不要な回答の削除ができます。

### データモデル（Realtime Database）
イベント定義と回答を **別ツリー** に分離し、回答は1件ずつ独立ノードに保存します。これにより「1回答の保存」が他の回答やイベント全体に触れず、同時更新でデータが失われません。

```jsonc
{
  "events": {
    "<eventId>": {
      "title": "忘年会の日程",
      "note": "場所未定",
      "ownerUid": "<作成者の匿名 uid>",     // ← 管理者の判定に使う（サーバー側で強制）
      "schemaVersion": 2,
      "createdAt": 0,                        // serverTimestamp
      "updatedAt": 0,
      "options": {
        "<optionId>": { "date": "2026-07-20", "time": "19:00", "order": 1 },
        "<optionId>": { "date": "2026-07-21", "order": 2 }   // time は任意
      }
    }
  },
  "responses": {
    "<eventId>": {
      "<responseId>": {                      // ← 名前ではなくランダム ID をキーにする
        "name": "田中",
        "authorUid": "<回答者の匿名 uid>",   // ← 本人判定に使う（サーバー側で強制）
        "answers": { "<optionId>": true },   // optionId をキーに ○(true)/✕(false)
        "createdAt": 0,
        "updatedAt": 0
      }
    }
  }
}
```

設計上のポイント:
- **候補は `optionId` で識別**し、回答は `answers[optionId]` で保存。候補の追加・削除・並べ替えをしても既存回答が別の日にずれません。新規追加した候補は既存参加者にとって「未回答（－）」であり、勝手に ○ になりません。
- **日付は `YYYY-MM-DD`、時間は `HH:mm` を素データとして保存**（`7/20(月)` のような表示文字列は保存しません）。表示時に `2026/7/20(月) 19:00` の形式へ整形します。日付は `new Date(str)` を使わず数値分解でパースし、タイムゾーンによる日付ずれを防いでいます。
- **回答の識別子は `responseId`（ランダム）**。同姓同名でも別々に扱われ、名前は重複可能です。

## セキュリティモデル

> Firebase の Web 設定（apiKey など）は **公開されて当然** の値です。安全性は「値を隠すこと」ではなく **Firebase Authentication ＋ Security Rules** で担保します。

### 採用方針：匿名認証（Anonymous Auth）＋ Security Rules
- 追加のサーバー（Cloud Functions）や課金プラン（Blaze）は不要で、GitHub Pages（静的配信）＋ Realtime Database だけで完結します。
- すべての訪問者はページ読み込み時に **匿名サインイン** し、サーバー発行の推測不可能な `uid` を得ます。永続化は端末ローカル（`browserLocalPersistence`）なので、同じブラウザなら再訪しても同じ `uid` を維持します。
- **管理者 = イベントの `ownerUid` と自分の `uid` が一致する人**。**回答の本人 = 回答の `authorUid` と自分の `uid` が一致する人**。この判定を **Security Rules がサーバー側で強制** します（UI でボタンを隠すのは補助にすぎません）。

### なぜ「秘密の管理者リンク / 編集トークン URL」を廃止したか
旧実装は管理者鍵のハッシュ（`h`）を **誰でも読めるレコード内に保存** し、権限判定を **クライアント JS だけ** で行っていました。REST を直接叩けば判定を回避でき、実質的に無防備でした。
URL に埋め込むベアラートークン方式も、Realtime Database の Rules では **トークンを安全に検証できません**（Rules はハッシュ計算ができず、照合のためにトークンを可読にすると秘密でなくなる）。そこで、**サーバー発行の `uid`（匿名認証）に権限を紐づける方式** に変更しました。これは URL 経由でのトークン漏洩（履歴・リファラ等）も避けられ、より安全です。

### ルールの要点（`database.rules.json`）
- ルート既定は `read/write: false`（明示的に許可した経路以外は全拒否）。
- `events/$eventId`
  - `read`: `auth != null`（＝匿名も含む認証済みのみ。未認証の REST 一括スクレイピングを遮断）
  - `write`: **作成時は `newData.ownerUid === auth.uid`、以後は既存 `ownerUid === auth.uid` のときだけ**（＝所有者のみ編集・削除可）
  - `ownerUid` は自分以外に書き換え不可、`$other` は拒否、`title/note/options/date/time/order` を型・書式・長さで検証
- `responses/$eventId/$responseId`
  - `read`: `auth != null`
  - `write`: **本人（新規は `authorUid===auth.uid`、既存は `authorUid===auth.uid`）** または **イベント所有者（管理者）** のみ
  - `authorUid` は後から書き換え不可、`answers` は真偽値のみ、`$other` は拒否
- したがって **他人の回答の改ざん・削除、他人のイベントの編集・削除はサーバー側で拒否** されます。

## Firebase セットアップ（コンソール作業）

> **あなたが手動で行う必要がある作業** です。

1. **プロジェクト**: 既存の `taku-f8db6` を使う場合はそのまま。別プロジェクトを使う場合は新規作成。
2. **Realtime Database を作成**: コンソール → 「Realtime Database」→「データベースを作成」→ ロケーションを選択 →（後述のルールを入れるので）**ロックモード** で開始。
3. **匿名認証を有効化（必須）**: コンソール → 「Authentication」→「Sign-in method」→ **「匿名」を有効化**。
   - 有効化しないと全員がサインインできず、`read/write` がすべて拒否され **アプリが動作しません**。
4. **Web アプリの設定を取得**: コンソール → 「プロジェクトの設定」→「全般」→「マイアプリ」→ Web アプリの構成（`apiKey`, `authDomain`, `databaseURL`, `projectId`, `appId`）。
5. **`js/firebase.js` の `firebaseConfig` を自分の値に置き換え**:
   ```js
   const firebaseConfig = {
     apiKey: 'REPLACE_WITH_YOUR_WEB_API_KEY',   // ← 置き換える
     authDomain: 'taku-f8db6.firebaseapp.com',
     databaseURL: 'https://taku-f8db6-default-rtdb.firebaseio.com',
     projectId: 'taku-f8db6',
     appId: 'REPLACE_WITH_YOUR_APP_ID',         // ← 置き換える
   };
   ```
6. **（推奨）承認済みドメインの追加**: Authentication →「Settings」→「承認済みドメイン」に `taku20200824.github.io` と `localhost` を追加。

**Cloud Functions は不要** です（匿名認証＋Rules で要件を満たすため）。

## Security Rules のデプロイ

Firebase CLI を使います（`database.rules.json` をプロジェクトに反映）。

```bash
# 1) CLI をインストール（未導入の場合）
npm install -g firebase-tools

# 2) ログイン
firebase login

# 3) プロジェクトを指定（初回のみ。プロジェクト ID は自分のものに）
firebase use --add          # 一覧から taku-f8db6 を選び、エイリアス default を付与

# 4) ルールだけをデプロイ
firebase deploy --only database
```

> ⚠ **デプロイの順番に注意**: 新ルールを反映すると、旧クライアント（未認証 REST）は読み書きできなくなります。**新しいクライアント（この `index.html`）の公開とルールのデプロイはできるだけ同時に** 行ってください。

コンソールから貼り付けても構いません: Realtime Database →「ルール」タブに `database.rules.json` の内容を貼り付け →「公開」。

## ローカル開発

ES Modules は `file://` では動きません。**正しい MIME タイプで配信する HTTP サーバー** が必要です（GitHub Pages は問題ありません）。

```bash
# 例: Node の serve（任意の静的サーバーで可）
npx serve .
# → http://localhost:3000 などで開く
```

ローカルでも Firebase の匿名認証・Rules は本番と同じものが使われます（`localhost` を承認済みドメインに追加しておくこと）。ルールや同時更新を安全に試したい場合は **Firebase Emulator Suite** を利用できます（`firebase.json` にポート設定済み）:

```bash
firebase emulators:start   # Auth: 9099 / Database: 9000 / UI 付き（Java 11+ が必要）
```

## 3種類の「リンク／権限」の違い
本アプリでは **URL は1種類（共有リンク）だけ** で、権限は URL ではなく **端末の匿名認証 ID** に紐づきます。

| 区分 | 誰か | 何ができるか | どう決まるか |
|---|---|---|---|
| **参加者リンク** | 共有 URL を開いた全員 | イベント閲覧・自分の回答の登録／編集／削除 | 共有 URL `…/#e/<eventId>`（全員共通） |
| **回答編集（本人）** | その回答を登録した端末 | 自分の回答のみ編集・削除 | 回答の `authorUid` ＝自分の `uid` |
| **管理者** | イベントを作成した端末 | 候補の編集、全回答の編集・削除 | イベントの `ownerUid` ＝自分の `uid` |

> 旧実装のような **秘密の管理者 URL / 編集トークン URL は存在しません**（前述の理由により、より安全な ID ベースへ移行）。管理者権限は「作成した端末（ブラウザ）」に紐づくため、別端末・別ブラウザ・データ消去後は管理者として扱われません（[残存リスク](#既知の制限残存リスク)参照）。

## 旧データ（レガシー）互換性
旧バージョンは 1 レコードに全状態を持つ形式（`{ t, d, o:[表示文字列], p:[回答], h:管理者鍵ハッシュ }`）でした。

- **後方互換リーダーを実装**しており、旧形式のイベントも **閲覧のみ** 表示できます（「📦 旧バージョン（閲覧のみ）」バッジと注意書きを表示）。
- 旧・URL 埋め込み形式（`#<base64>`）のリンクも **閲覧のみ** で開けます。
- 旧イベントを **破壊・削除しません**。新ルールでは旧形式への書き込みは拒否されるため、実質的に読み取り専用アーカイブになります。
- **自動的な管理者移行は行いません**。旧管理者鍵はクライアント側ハッシュでサーバー検証できないため、これを新しい管理者権限へ引き継ぐと「無関係な人へ管理者権限を与える」危険があります。よって旧イベントの管理権限は移行されません。
- 旧イベントで回答収集を続けたい場合は、**「新規作成」で作り直す**運用を案内しています（旧候補は表示文字列のみで正確な `YYYY-MM-DD` を復元できないため、候補の自動移行は行いません）。

## バリデーションと上限
クライアントと Security Rules の両方で検証します（`js/model.js` の `LIMITS` とルールが対応）。

| 項目 | 上限・形式 |
|---|---|
| タイトル | 1〜100 文字 |
| メモ | 0〜2000 文字 |
| 参加者名 | 1〜40 文字 |
| 候補数 | 1〜60 件 |
| 回答数 | クライアント側の目安 500 件（Rules では件数上限は課さない。下記制限参照） |
| 日付 | `YYYY-MM-DD`（実在日のみ。`2026-02-30` 等は拒否） |
| 時間 | `HH:mm`（任意）|
| ID（event/response/option）| `[A-Za-z0-9_-]{6,64}` |

ユーザー向けメッセージは状況別に出し分けます: **イベントが見つからない／権限がない／オフライン／通信エラー／不正な入力／（レガシーで）閲覧のみ**。

## 既知の制限・残存リスク
- **管理者・回答編集の権限は「端末（ブラウザ）」に紐づく**。ブラウザデータを消去する、別端末・別ブラウザから開く、シークレットウィンドウで開くと、同一人物でも管理者／本人として扱われません。複数端末で管理したい場合は Cloud Functions によるカスタムトークン発行や、匿名アカウントのメール連携（アカウントアップグレード）が必要です（本実装では未対応）。
- **リンクを知っている人は誰でも閲覧・回答できます**（日程調整アプリの性質上の仕様）。閲覧を限定したい用途には不向きです。
- **回答件数の上限は Rules では強制していません**（RTDB ルールで子要素数を数えるのは困難なため）。大量投稿対策が必要なら Cloud Functions や App Check の導入を検討してください。
- **匿名認証の濫用対策**として、本番では [Firebase App Check](https://firebase.google.com/docs/app-check) の併用を推奨します。
- レガシーイベントは **閲覧のみ**（前述）。

## テスト
- **ロジック単体テスト（Node）**: `js/model.js` の日付処理（UTC ずれ防止）、バリデーション、`optionId` による回答保持（追加・削除・並べ替えで回答がずれない／新候補が自動 ○ にならない）、レガシー変換を検証。
- **ブラウザ結合テスト**（Firebase をインメモリのモックに差し替え、`app.js`/`event.js`/`ui.js` をそのまま実行）で以下を確認:
  - 2つのブラウザからの同時回答が互いを上書きしない（別 `responseId` へ独立保存）
  - 同姓同名の参加者が別々に登録できる
  - 候補の追加・削除で既存回答が保持され、新候補は「未回答」
  - 非管理者はイベント編集・他人の回答編集／削除ができない（`PERMISSION_DENIED`）
  - 本人は自分の回答のみ編集・削除できる／管理者は全回答を管理できる
  - オフライン→オンラインの状態遷移
  - モバイル幅・ダークモードでの表示
  - レガシーイベント／レガシー URL を開いたときの読み取り専用表示
  - ユーザー入力の HTML/スクリプトがそのまま表示（XSS が発火しない）
  - 保存ボタン連打で重複データが作られない

> リアルタイム性・Rules の実強制・実ブラウザ間の同時更新は、**Firebase Emulator Suite** もしくは本番プロジェクト（匿名認証を有効化しルールをデプロイ済み）での手動確認を推奨します。
